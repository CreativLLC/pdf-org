# `sf-integration-change-execute`

You are implementing the integration change against the engagement repo's working tree, following the plan produced by `sf-integration-change-plan`. **You do not deploy, run tests, or update Jira here** — those are downstream steps.

## Inputs

- `$ARTIFACTS_DIR/plan.md` — the full plan written by the plan step
- `$plan.output` — the JSON summary
- `$load-engagement-context.output` — patterns/standards in scope
- `$verify-org-context.output` — org info (`api_version`, etc.)
- `$classify-sub-type.output` — `sub_type`, `external_system`, side flags

## Tools

Read, Edit, Write, Glob, Grep, Bash (for `git status`, `git diff` only — no commits). No SF CLI calls. No Jira writes. No network. No reads of `~/.archon/credentials/` (per the user memory invariant — credentials never enter the conversation context).

## Task

1. **Read the full plan** from `$ARTIFACTS_DIR/plan.md`. Treat it as authoritative.
2. **Implement the file changes** per the plan's `files_changed` list:
   - For `add` operations: Write the new file with content that satisfies the plan and adheres to every pattern in `patterns_followed`.
   - For `modify` operations: Edit the existing file with surgical changes. Preserve unrelated code formatting and structure.
   - For `delete` operations: Remove the file. If the engagement uses `source_tracked` dev model, also update `manifest/destructiveChanges.xml` (or create it if absent). If `org_development`, leave the destructive manifest update to the deploy step.
3. **Adhere to the patterns in scope.** Specifically:
   - **Every callout MUST** use `callout:<Named_Credential>/<path>` syntax. NEVER hardcode a URL. NEVER paste a token, OAuth secret, or API key into Apex code. The credentials hygiene check WILL fail if any of these appear.
   - **Every callout MUST** set an explicit `setTimeout(<ms>)` on the `HttpRequest`. The pattern compliance check WILL fail if absent.
   - **Every callout MUST** return a structured `Result` wrapper (or equivalent class with `success`, `statusCode`, `body`, `errorMessage`, `isTransient` fields). NEVER return raw `HttpResponse` to callers. See [`patterns/apex-callout-pattern.md`](../patterns/apex-callout-pattern.md) §"The service class."
   - **Every callout MUST** distinguish transient errors (5xx, `CalloutException` for network) from permanent errors (4xx). The Result's `isTransient` field carries this signal; callers retry transient and log permanent.
   - **Callouts after DML** MUST be wrapped in a Queueable per [`patterns/queueable-async-pattern.md`](../patterns/queueable-async-pattern.md). A callout directly from a trigger fails at runtime with `CalloutException: Callout from triggers are currently not supported`.
   - **Webhook handlers (`@RestResource`) MUST** include an authentication check. Acceptable forms: (a) Connected App OAuth via session check, (b) explicit IP allowlist read from a documented Custom Metadata Type, (c) signature verification via HMAC against a secret stored in a Named Credential, (d) signed JWT verification. NEVER ship a webhook handler that accepts unauthenticated POSTs.
   - **Platform Event publishers** use `EventBus.publish` synchronously (returns `Database.SaveResult[]`). Subscribers (Apex triggers on `<PE>__e`, Flows) must be updated in lockstep when fields change — the pre-execute gate displayed the subscriber list; honor it.
   - **Named Credentials** declare endpoint + auth declaratively in XML. Tokens for `Password` protocol are entered manually in Setup once the NC is deployed — they NEVER appear in source.
   - **Connected Apps** declare OAuth config in XML. The consumer secret is NEVER in source; it's generated by Salesforce on deploy.
4. **Credential reference invariants** per [ADR-0008](../decisions/0008-credential-management.md):
   - Apex strings of the form `'sk_live_...'`, `'sk_test_...'`, `'xoxb-...'`, `'xoxp-...'`, `'ghp_...'`, `'AKIA...'`, `'ya29...'`, 40+ char hex literals, or any other obvious secret prefix are forbidden in source. The credentials hygiene check WILL fail.
   - The Apex pattern is `req.setEndpoint('callout:<NC_Name>/<path>');` — the runtime injects auth from the Named Credential. Do NOT construct `Authorization` headers with literal values.
   - For HMAC signature verification, the secret is read from a Named Credential (`Stripe_Webhook_Signing_Secret` style) at runtime via `NamedCredential` helper APIs. The secret never appears in Apex source.
   - If the plan calls for a new Named Credential, the XML declares endpoint + auth scheme. The actual token entry happens manually in Setup after deploy — record this as a step in `$ARTIFACTS_DIR/manual-setup-steps.md` so the engineer knows what to do post-merge.
5. **If you encounter a situation the plan didn't anticipate**, **add the variation to the plan first** (write a new section to `$ARTIFACTS_DIR/plan.md` describing the deviation and why), THEN implement.
6. **Do not modify unrelated files.** If you find an issue elsewhere (a bug in a different class, a stale doc, a typo), record it in `$ARTIFACTS_DIR/follow-ups.md` for a separate ticket — don't fix it here.
7. **Stage the changes for review.** Run `git status` and capture the file list. Do NOT `git add` or `git commit` — the engineer commits after the workflow completes successfully.
8. **Write an implementation summary** to `$ARTIFACTS_DIR/implementation.md` describing:
   - Files actually changed (vs. what the plan predicted)
   - Any plan deviations and their justification
   - Manual setup steps required post-merge (NC token entry, Connected App consumer key/secret retrieval, Platform Event subscriber assignment in Setup, etc.)

## Output

Emit a structured JSON summary on stdout:

```json
{
  "files_changed_actual": [
    {"path": "force-app/main/default/classes/StripeBillingService.cls", "operation": "add", "lines_added": 84, "lines_removed": 0},
    {"path": "force-app/main/default/classes/StripeBillingService_Test.cls", "operation": "add", "lines_added": 62, "lines_removed": 0}
  ],
  "named_credentials_referenced": ["Stripe_API"],
  "plan_deviations": [],
  "follow_ups_recorded": false,
  "manual_setup_steps_required": true,
  "implementation_artifact": "$ARTIFACTS_DIR/implementation.md",
  "manual_setup_artifact": "$ARTIFACTS_DIR/manual-setup-steps.md"
}
```

The model used for this node is `opus[1m]` (per the workflow YAML) — integration code is the heaviest reasoning step in this family because it interacts with three concerns simultaneously: the Apex pattern, the metadata XML, and the credential hygiene invariants. Take the full context budget.
